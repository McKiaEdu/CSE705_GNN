## Purpose

Defines what gets run and runs it. Enumerates the configuration grid, constructs the
model / hooks / readout / metrics objects for each configuration, calls `TrainRun`, and
serializes the returned record to `results/`. It is the only component that knows the
shape of the experiment, and the only one that writes to disk.

Realizes D-022 (single writer), D-023 (hiddenDim), and D-027 through D-031 below.

## Approach (agreed)

**A declarative grid plus a dumb runner.** The sweep is a list of `RunConfig` objects
generated by pure functions, and the runner iterates that list calling `TrainRun`. No
configuration logic lives in the loop; nothing is decided at run time. This makes the
grid inspectable — the run count can be printed and checked before anything executes,
which matters when the total is in the hundreds.

The decisions this carries, and what each rejects:

**Depth grid `{2, 4, 8, 16, 32}`, log-spaced.** Cai & Wang's contraction bound makes
energy decay exponential in depth, so log-spaced depths are evenly spaced on the
log-energy axis that every headline figure uses.
*Rejected — adding 24.* Buys resolution between two points where the curve has already
collapsed, and breaks the log spacing. The fine structure of the energy curve comes from
the layer index within a run, not from additional depth points.

**Seeds are the literal list `0..9`.** Written into the record and the README so a
reproduction attempt uses the same ten, not ten drawn from a meta-RNG whose state is
another thing to record.

**`"jk"` stays in `config.mitigations` despite being a readout.** It is redundant with
`config.readout` (D-016), and the redundancy is accepted rather than removed: the results
filename is built from the mitigation list, `mitigations = []` renders as `none`, so
dropping `"jk"` would collide a JK run with the baseline at
`gcn_none_d8_s0.json`. Removing the redundancy would require adding the readout to the
filename, which is a contract change for no benefit.
*Guard:* `("jk" in mitigations) == (readout == "jk")` is asserted at write time, so the
two fields cannot silently disagree.

**The hyperparameter search is small and exists to justify, not to optimize.** Eight
configurations on GCN at depth 2, three seeds. Tuning at depth 2 and holding the winner
fixed across every architecture and depth is the methodologically correct choice: tuning
per depth would confound the depth effect with a hyperparameter effect, exactly as C3
warns.
*Selection rule, three levels, fully specified so it is not decided ad hoc when the
numbers arrive.* Primary: highest mean validation accuracy, since accuracy is the metric
the report leads with. Tie-break: where the top configurations fall within one standard
deviation of each other, take the lower mean validation loss, which is the finer signal
for exactly the reason D-017 gives — accuracy over 500 validation nodes moves in steps of
0.002. Final fallback: where loss also ties, take the configuration closest to the
published baseline (learning rate 0.01, dropout 0.5, weight decay 5e-4) and say so in the
report.
*Note on the apparent tension with D-017.* D-017 selects an epoch within a run using
validation loss; this selects a configuration across runs using validation accuracy.
Different operations on different objects — but the coarseness objection D-017 raises
applies here too, which is why loss is the tie-break rather than absent.
*Rejected — a per-architecture or per-depth search.* Multiplies the grid and destroys
the comparison the study is built on.

**The sweep is idempotent.** A run whose results file already exists is skipped unless
explicitly forced. At this run count on a single machine over a two-week window, the
sweep will be interrupted; restarting from zero each time is the failure mode that eats
the schedule.

## Interface & contract touchpoints

Functions/methods use PascalCase; variables use camelCase; PyG `Data` fields keep their
library names (`edge_index`, `train_mask`).

- `class RunConfig` — defined here; frozen dataclass fully describing one run:
  `convType: str`, `numLayers: int`, `mitigations: list[str]`, `readout: str`,
  `hiddenDim: int`, `dropout: float`, `learningRate: float`, `weightDecay: float`,
  `maxEpochs: int`, `patience: int`, `seed: int`, `saveEmbeddings: bool = False`.
- `BuildGrid(arm: str) -> list[RunConfig]` — defined here; pure, no side effects.
  Enumerating the grid must be possible without running anything.
- `BuildModel(config: RunConfig) -> GnnModel` — defined here; constructs the conv stack,
  the `layerHooks` list in D-007 order, and the readout (`LastLayerReadout` or
  `JkReadout`).
- `RunOne(config: RunConfig, data: Data, metrics: OversmoothingMetrics) -> dict` —
  defined here; builds the model, calls `TrainRun`, returns the record.
- `RunSweep(arms: list[str], resultsDir: str, force: bool = False) -> None` — defined
  here; iterates, skips existing, writes.
- `ResultPath(config: RunConfig, resultsDir: str) -> str` — defined here; the single
  place the C3 filename convention is expressed.
- consumes: `LoadCora` from `data/`; `GnnModel` and `LastLayerReadout` from `models/`;
  `ResidualHook`, `PairNormHook`, `JkReadout`, `MitigationNames` from `mitigations/`;
  `OversmoothingMetrics` from `metrics/`; `TrainRun` and `TrainConfig` from `train/`.
- produces: `results/<convType>_<mitigations>_d<depth>_s<seed>.json`, one per run,
  conforming to C3; and `results/embeddings/*.pt` for the flagged runs of D-031.

Contract properties this component must honor:

- **It is the only writer** (D-022). `train` returns a record; nothing else touches
  `results/`.
- **`ResultPath` is the sole expression of the filename convention.** Any other module
  needing a path calls it.
- **The `("jk" in mitigations) == (readout == "jk")` assertion fires before write**, not
  after.
- **`saveEmbeddings` is excluded from the serialized `config` block.** It directs the
  runner, not the model, and C3 defines no such field. `RunConfig` therefore maps
  one-to-one onto C3's `config` block across all fields except this one, and two runs
  differing only in `saveEmbeddings` produce identical records.
- **`OversmoothingMetrics` is constructed once per run**, not once per sweep, since it
  holds the augmented operator as state and `metrics` asserts self-loop-free input on
  construction (D-004).
- **`hiddenDim = 64` everywhere except arm E** (D-023).

## Implementation plan

- `BuildGrid(arm)` — defined here. Six arms, each a pure enumeration:

  - **A — depth sweep, unmitigated.** `convType` in {gcn, sage, gat} x depth in
    {2,4,8,16,32} x seed in 0..9. `mitigations = []`, `readout = "lastLayer"`.
    **150 runs.**
  - **B — mitigation ablation on GCN.** `mitigations` in {["residual"], ["pairnorm"],
    ["jk"], ["pairnorm","residual"]} x depth x seed. **200 runs.**
  - **C — GCNII across depth.** `convType = "gcnii"`, `mitigations = []`, depth x seed.
    GCNII is a convType, not a mitigation (D-008), but appears in the mitigation
    comparison figure. **50 runs.**
  - **D — best mitigation carried across architectures.** `convType` in {sage, gat} x
    depth x seed, with the single mitigation arm B found most effective. This arm is
    generated only after B has been aggregated; the winner is passed in rather than
    hardcoded. **100 runs.**
  - **E — fidelity arm.** `convType = "gcn"`, depth 2, `hiddenDim = 16`, seed 0..9. The
    published-baseline reproduction of D-023. **10 runs.**
  - **F — hyperparameter search.** GCN at depth 2, `learningRate` in {0.005, 0.01} x
    `dropout` in {0.5, 0.6} x `weightDecay` in {5e-4, 1e-3} x seed in 0..2. Selected by
    the three-level rule above; the winner is frozen into the defaults used by A–E.
    **24 runs.**

  **534 runs total.** Order of execution: F first (it sets the defaults), then A, B, C,
  E, then D once B is aggregated.

  `saveEmbeddings` is set True for ten of those configurations per D-031:
  `convType = "gcn"`, `seed = 0`, depth in {2, 32}, for each of the five mitigation arms
  ([], ["residual"], ["pairnorm"], ["jk"], ["pairnorm","residual"]). These are existing
  arm A and arm B configurations with a flag set — no additional runs, and the totals
  above are unchanged.

- `BuildModel(config)` — defined here. Constructs `layerHooks` in D-007 order (residual
  before PairNorm) and selects the readout from `config.readout`. Reads GCNII's
  `alpha = 0.1` and `lambda = 0.5` from module constants, not from `RunConfig`, since
  they apply to one convType only and would be null fields on every other run.
- `RunOne(config, data, metrics)` — defined here. Builds the model, calls
  `TrainRun(model, data, TrainConfig(...), metrics)`, returns the record. Does not write.
- `RunSweep(arms, resultsDir, force)` — defined here. For each config: compute
  `ResultPath`; if it exists and not `force`, skip and continue; else `RunOne`, assert
  the jk/readout consistency, serialize. Prints a running count and the skip count so an
  interrupted sweep reports what remains.
- `ResultPath(config, resultsDir)` — defined here. `<convType>_<mitigations>_d<depth>_s<seed>.json`
  with `<mitigations>` the sorted list joined by `+`, or `none` when empty.
- `AggregateArm(resultsDir, groupBy) -> ...` — NOT defined here. Aggregation and
  plotting belong to `viz/`; `experiments` produces records and stops.

Vectorization is not a concern in this module — it contains no numerical work. The loops
are over configurations and are inherently sequential.

## Dependencies

- Depends on: every other module — `data/`, `models/`, `mitigations/`, `metrics/`,
  `train/`. This is the top of the dependency graph.
- Consumed by: `viz/` reads the records it writes, but does not import it.
- Build order: last of the executable modules, before `viz/`.

## Assumptions & constraints

Every item here is provisional until confirmed.

- Python / version: Python 3.12.13.
- Libraries / role: `torch`, `torch_geometric` 2.8.0. No orchestration framework — a
  plain loop, since the sweep is sequential and single-machine.
- Compute: CPU wheels, device-agnostic. **Wall-clock is deliberately not estimated in
  this spec.** The depth-32 runs dominate, and MAD over 33 layers at 3 captures is the
  per-run cost driver. Time one depth-32 GCN run before committing to the full grid; that
  number times ~150 determines whether the grid fits the remaining schedule. If it does
  not, the reduction to make first is seeds 10 -> 5 on arms B and D, which halves 300 runs
  while leaving arm A's headline result at full seed count.
- `hiddenDim = 64` (D-023) on every arm except E.
- GCNII constants: `alpha = 0.1`, `lambda = 0.5`, matching Chen et al.'s published Cora
  configuration (arXiv:2007.02133, Table 6, verified by fetching).
- Confirmed assumptions:
  - Seeds are the literal integers 0 through 9.
  - Arm D's mitigation is chosen from arm B's aggregated results, not fixed in advance.
  - Arm F's winner is frozen before arms A–E run, so every reported run shares one
    hyperparameter configuration.
  - `NormalizeFeatures` and the public split are fixed for every run (D-002).

## Outputs & artifacts

- `results/<convType>_<mitigations>_d<depth>_s<seed>.json` — 534 files at full grid, one
  per run, each conforming to C3.
- `results/embeddings/<runId>_l<index>.pt` — 16 tensors from the ten flagged runs,
  roughly 11 MB total, consumed only by `viz`. Two per flagged run (the first and last
  band indices), except where those coincide — depth-2 runs under `LastLayerReadout`,
  whose band is the single point `1..L-1` — which save one. The depth-2 JK run has a
  two-point band `1..L` and saves two. Gitignored.
- A printed sweep summary at completion: total configurations, runs executed, runs
  skipped as already present, runs failed. Written to stdout, not to a file — the
  records are the artifact.
- `results/` is gitignored except for a `.gitkeep`; the README documents that the sweep
  regenerates it.

## Test plan

- **Grid arithmetic:** `len(BuildGrid(arm))` equals 150 / 200 / 50 / 100 / 10 / 24 for
  arms A–F. Asserted, because a silent off-by-one in an enumeration is invisible until
  the aggregation has a hole in it.
- **Grid purity:** `BuildGrid` called twice returns equal lists, and calling it performs
  no I/O and constructs no model.
- **No duplicate paths:** across the whole grid, `ResultPath` values are unique. This is
  the regression test for the `"jk"` naming decision — with `"jk"` dropped from
  `mitigations`, this test fails on a collision with the baseline.
- **jk/readout consistency:** constructing a `RunConfig` with `mitigations = ["jk"]` and
  `readout = "lastLayer"` fails the assertion.
- **Embedding flags:** across the full grid, exactly ten configurations have
  `saveEmbeddings == True`, all with `convType == "gcn"`, `seed == 0`, and depth in
  {2, 32}, covering all five mitigation arms at each depth. A miscounted flag is
  invisible until the t-SNE figure has no input.
- **Embedding tensor count:** after running the ten flagged configurations,
  `results/embeddings/` contains exactly 16 files. This is where the derived-band
  decision (D-001 C1) shows up in the file system, and the count differing from 20 is
  correct rather than a shortfall.
- **`saveEmbeddings` is not serialized:** two `RunConfig` values differing only in
  `saveEmbeddings` produce byte-identical `config` blocks in their records.
- **Idempotency:** run a two-config sweep twice; assert the second pass executes zero
  runs and reports two skips, and that file mtimes are unchanged.
- **Force overrides skip:** the same sweep with `force = True` re-executes both.
- **Record conformance end to end:** for one cheap config (GCN, depth 2, seed 0), run the
  full path and validate the written JSON against C3's key set, including that
  `config` round-trips to an equal `RunConfig` (excepting `saveEmbeddings`).
- **Hook order:** for `mitigations = ["pairnorm","residual"]`, assert `BuildModel`
  produces `layerHooks` ordered residual-then-PairNorm regardless of the sorted list
  order in the config (D-007).
- **Arm E width:** every arm E config has `hiddenDim == 16`; every other arm has 64.
- **Interrupt safety:** kill a sweep mid-run and restart; assert no partial or corrupt
  JSON file exists. Write to a temporary path and rename, so a file is either complete or
  absent.

## Report / novelty note

This module supplies Section 4's experimental protocol and the reproducibility claim the
README makes. Three points for the writeup:

1. **The grid is stated as arithmetic, not prose.** 534 runs decomposed by arm, with the
   seed and depth sets given explicitly, is what makes the results section checkable
   rather than assertable.
2. **The hyperparameter configuration is coordinated, and that is a methodological
   choice.** One configuration tuned on GCN at depth 2 and held fixed across every
   architecture and depth. It is not laziness — varying hyperparameters per depth would
   confound the depth effect the entire study measures. State it as a choice with its
   reason, and state its cost: GAT may be disadvantaged, since it publishes at a lower
   learning rate.
3. **The fidelity arm converts a deviation into a measurement.** Arm E exists solely so
   the report can give the published-baseline number at Kipf & Welling's own width
   alongside the sweep's number at 64, rather than asserting that the width change is
   immaterial.

## Open questions

- **Wall-clock feasibility.** Unresolved by design. Measure one depth-32 run before
  committing. The pre-agreed reduction, if needed, is seeds 10 -> 5 on arms B and D only.
- **Arm D's mitigation is not yet known**, since it depends on arm B's aggregate. The
  grid function takes it as an argument; the value is filled in after B completes.
- **Failure handling.** If a single run raises — a `nan` loss at depth 32, say — does the
  sweep abort or record the failure and continue? Recording and continuing is more useful
  at this run count, but it needs a failure marker in `results/` so the gap is visible at
  aggregation rather than looking like a run that was never scheduled. Decide before the
  first full sweep.
- **GCNII weight decay.** Chen et al. use 0.01 on convolutional layers and 5e-4 on dense
  layers; D-019 holds a uniform 5e-4 for comparability. If GCNII underperforms its
  published depth behavior, this is the first suspect. Whether to run a single
  GCNII-at-published-decay diagnostic is undecided.
- **Whether arm F's three seeds are enough** to separate eight configurations whose
  validation accuracies may differ by less than seed noise. The three-level selection
  rule makes the tie case deterministic rather than leaving it to judgment, but if every
  configuration ties at the first two levels, that is itself worth one sentence in the
  report: the study is insensitive to these hyperparameters in this range.
